## Session — 2026-04-14

### 1. What we did / learned

Built and fully specified a linear programming model to answer: given a foraging time budget and character build, which Skyrim regions are worth your time for alchemy?

**Data pipeline:**
- Scraped 191 ingredients from UESP MediaWiki API (fetch_data.py), parsing `{{Ingredient Summary}}` templates for effects and location sections for regional spawn counts.
- Derived potion recipes from shared-effect ingredient pairs (potions.json, ~5,238 total combinations).
- Normalized yield rates globally: yield = spawn_count / 503 × 4.0, where 503 is Dragon's Tongue in Winterhold/Eastmarch — the highest single-region count of any ingredient. Earlier per-ingredient normalization was making rare ingredients (Human Flesh: 14 spawns) appear as abundant as common ones.

**LP formulation (final, correct):**
- Decision variables: x[r] = hours in region r; z[pair] = batches brewed using a specific ingredient pair.
- Objective: maximize Σ weight[potion_name] × z[pair].
- Ingredient conservation constraint: Σ_{pairs using ingredient i} z[pair] ≤ Σ_r x[r] × yield[r][i]. One constraint per unique ingredient. This is the critical constraint — it prevents the same unit of an ingredient from being claimed simultaneously by multiple recipes.
- Time budget: Σ x[r] ≤ T.
- Solver: scipy.optimize.linprog with HiGHS backend. 6 region variables + up to ~300 pair variables per solve (build-filtered).

**Multi-pair LP (the key modeling decision):**
The primary LP considers all 1,175 valid ingredient pairs (pairs where both ingredients have yield data and the potion has nonzero weight for the build). The solver picks the optimal mix — it is not locked into one canonical recipe per potion type. This materially changed the results: Hjaalmarch / The Pale became the dominant region for 4 of 6 builds, and the Pure Mage's allocation reversed entirely (Whiterun → The Reach, via Grass Pod + Wild Grass Pod for Restore Magicka).

**Three-ingredient secondary LP:**
Given the ingredient totals produced by the primary foraging LP, a secondary LP finds the optimal mix of two-ingredient and three-ingredient brews. Only "pure" three-ingredient combos (all activated effects same polarity) are eligible — mixed-polarity brews hurt the wrong side. Gains above the two-ingredient baseline range from 2–11% across builds. These gains are modest because the primary LP already captures ingredient flexibility through pair selection; the incremental value of specifically combining three ingredients is small.

**Blog post additions:** Three-ingredient analysis section, ingredient conservation explanation embedded in the model description, updated caveat in "What the model doesn't know."

### 2. How we did it

- Python + scipy.optimize.linprog (HiGHS backend) for all LP solves. Runs in milliseconds even with ~300 pair variables.
- `enumerate_pure_triples()` in analyze.py iterates all C(n,3) ingredient combinations (~14K), filters on polarity purity and build relevance, and returns a ranked list by synergy_gain = triple_weight − best_pair_weight.
- `stress_test_pair_selection()` in analyze.py runs the full-enumeration foraging LP (all valid pairs) and compares objective and regional allocation (L1 distance) against the canonical-pair LP. Run with `--stress-test` flag.
- `brewing_allocation_lp()` runs the secondary LP, solving twice (two-ingredient only, then mixed) and returning objective comparison.
- Charts generated by matplotlib, blog-styled, saved to `public/images/skyrim-alchemy/`.

### 3. What worked / what didn't

**Worked:**
- Global normalization correctly de-prioritized rare ingredients. Human Flesh went from yield 4.0 (the maximum) to ~0.032.
- The conservation constraint fix was clean — one change to the supply constraint block in `build_and_solve()`, from per-potion-per-ingredient bounds to per-ingredient shared bounds.
- The stress test answered two questions at once: it exposed the conservation flaw and (after fixing it) confirmed that canonical pair selection does significantly alter the advice.

**Didn't work / required correction:**
- **Per-ingredient normalization:** Early version normalized each ingredient to [0,4] independently, making rare ingredients appear as abundant as common ones. Fixed by global normalization.
- **Ingredient non-conservation in original LP:** The original supply constraints (`z[p] ≤ supply[i]` per potion per ingredient) allowed the same unit of an ingredient to be claimed by multiple recipes simultaneously. Inflated objective values by 300–700% in the full-enumeration stress test before the fix. Fixed by replacing with per-ingredient conservation constraints.
- **Canonical pair selection:** Choosing one pair per potion type (highest combined yield) was initially thought to be a minor simplification. Stress testing showed it meaningfully alters the regional allocation for most builds — most notably the Pure Mage, which was routed to the wrong region entirely (Whiterun vs. The Reach). Fixed by using all valid pairs.
- **Mixed-polarity three-ingredient combos:** Early enumeration incorrectly scored combos with positive and negative effects together. Skyrim's brew classification is determined by the highest-gold-value effect, meaning mixed combos either poison the player (negative effects in a potion) or buff enemies (positive effects in a poison). Fixed by polarity filter.

### 4. How to improve

- **Ingredient accessibility weighting.** The LP treats all ingredient pairs as equally accessible: a dungeon-placed item and an outdoor respawning plant are worth the same per spawn count unit. In practice, outdoor respawning plants are much faster to farm. A weighting factor (e.g. 0.2× for monster drops, 0.5× for placed indoor ingredients) would make the foraging advice more realistic.
- **Travel time.** The LP ignores travel between regions. A player without fast travel faces real costs when allocations spread across three regions. An adjacency penalty matrix could address this.
- **The "top N triples" truncation.** The secondary brewing LP only considers the top 50 triples by synergy gain. This is a practical cap; occasionally a lower-ranked triple might be more accessible geographically. A per-build filter on which triples are compatible with the primary LP's ingredient totals would be cleaner.

### 5. Automation opportunities

- **Stress-test runner.** `analyze.py --stress-test` already runs Test A (full-enum LP vs. multi-pair LP). Could be extended to a systematic "model sensitivity report" that runs across all formulation variants and outputs a structured comparison — useful if the model is revised again.
- **Ingredient accessibility tier file.** A JSON file mapping ingredient names to a farming difficulty tier (outdoor respawning, indoor placed, monster drop) would enable the accessibility weighting improvement above without touching the LP structure.

### 6. Recommendation on item 5

- **Stress-test runner:** Keep as-is (`--stress-test` flag). Not worth formalizing into a separate report pipeline until the model is at a stage where regular re-validation would be triggered (e.g. if UESP data is re-scraped).
- **Ingredient accessibility tier file:** Worth building before a second post or any model update that changes foraging advice. It's a data artifact, not code — could be partially generated from the existing ingredient data (monster-drop flag is already in the UESP data) and manually reviewed for the rest.
